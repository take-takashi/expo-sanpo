# 0001. Remote Codex CLI 操作に tmux を前提として使う

## 状態

提案中

## 背景

iPhone アプリから母艦の Mac に SSH 接続し、Mac 上で Codex CLI を操作したい。

iPhone アプリ側に汎用ターミナルエミュレータを実装すると、ANSI エスケープシーケンス、対話入力、画面差分、リサイズ、セッション維持などを扱う必要があり、初期実装の範囲が大きくなる。

一方で、SSH 接続先で tmux を使えば、Codex CLI の実行状態を Mac 側に保持でき、アプリ側はプロンプト送信と結果取得に集中しやすい。

## 判断

Remote Codex CLI 操作では、SSH 接続先の tmux セッション上で Codex CLI を起動する構成を前提にする。

ただし、実装詳細は未確定であり、`tmux capture-pane` などで十分に扱えるかは検証する。

## 理由

- SSH 接続が切断されても Codex CLI の実行状態を維持しやすい。
- iPhone アプリ側で汎用ターミナルエミュレータを作る必要を減らせる。
- 母艦の Mac に既存の開発環境、認証情報、Codex CLI 設定を集約できる。
- 最初のプロダクト価値を、ターミナル再現ではなく、散歩中のプロンプト投入と結果確認に集中できる。

## 影響

- tmux の導入と基本操作が Mac 側の前提条件になる。
- アプリは tmux セッションの作成、選択、再接続、出力取得を扱う必要がある。
- Codex CLI の状態判定は、tmux の出力取得だけでは難しい可能性がある。

## 代替案

| 代替案 | 採用しない理由 |
| --- | --- |
| iPhone アプリに汎用ターミナルエミュレータを実装する | 初期実装の範囲が大きく、今回の価値に対して重い |
| Mac 側に専用エージェントを常駐させて Codex CLI と連携する | 将来的には有力だが、初期段階では追加コンポーネントが増える |
| Codex CLI ではなく API で直接実装する | 母艦 Mac の既存環境を使う目的から外れ、今回の前提と異なる |

## 未解決事項

- Expo から SSH 接続を直接扱う実装が現実的か。
- TTS をアプリ側で行うか、Mac 側で行うか。
