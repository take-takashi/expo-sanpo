# Feasibility

## 結論

`expo-sanpo` は実現可能性がある。ただし、Expo Go だけで完結できる範囲は限定される。

最初の検証では、Expo Go で UI、プロンプト入力、結果表示、TTS の基本動作を確認する。一方で、iPhone アプリから SSH 接続を直接扱う実装は、Expo Go の制約に当たる可能性が高いため、早い段階で代替構成を検証する。

## Expo Go で検証しやすいこと

- React Native と Expo による iPhone アプリの画面構成。
- プロンプト入力と履歴表示の UX。
- Codex CLI の出力を想定したテキスト表示。
- `expo-speech` による基本的な TTS。
- 読み上げが無い場合のテキスト中心の開発指示体験。

## Expo Go で難しい可能性が高いこと

- iPhone アプリから SSH 接続を直接確立すること。
- SSH に必要なネイティブライブラリを追加すること。
- iOS のキーチェーンや秘密鍵管理を、SSH ライブラリと組み合わせて十分に検証すること。
- irodori-TTS など、Expo Go に含まれないネイティブ機能や外部ランタイムを組み込むこと。

Expo Go は固定されたネイティブライブラリを含むアプリであり、含まれていないネイティブコードを後から追加できない。そのため、SSH ライブラリや特殊な TTS 実装がネイティブコードを必要とする場合は、Development Build への移行が必要になる。

## SSH 直接接続と Mac 側ブリッジの比較

SSH 直接接続は、Mac へ入って tmux を操作するという目的には自然である。特に、通常の CLI やネイティブアプリで実装するなら、SSH のほうが構成要素は少ない。

一方で、Expo Go を使う初期検証では、SSH 直接接続の実装が難所になりやすい。Expo Go に含まれないネイティブ SSH ライブラリを使う場合は、その時点で Development Build が必要になる。

Mac 側ブリッジは、iPhone アプリから見ると通常の HTTP または WebSocket 通信になるため、Expo Go でも検証しやすい。ただし、Mac 側に小さな常駐プロセスを置く必要があり、認証、ポート公開、プロセス管理、セキュリティ設計が増える。

| 観点 | SSH 直接接続 | Mac 側ブリッジ |
| --- | --- | --- |
| 目的との素直さ | 高い | 中程度 |
| Expo Go での実装しやすさ | 低い可能性が高い | 高い |
| Development Build の必要性 | 高い可能性がある | 低い |
| Mac 側の追加実装 | 少ない | 必要 |
| iPhone 側の秘密鍵管理 | 必要 | 不要または軽くできる |
| セキュリティ設計 | SSH 鍵管理が中心 | ブリッジ認証と公開範囲が中心 |
| tmux 操作の自由度 | 高い | ブリッジ API の設計次第 |

初期検証では、Expo Go で動かすことを重視するなら Mac 側ブリッジが現実的である。最終形としては、SSH 直接接続のほうがシンプルになる可能性がある。

## 現実的な検証ルート

### Phase 1: Expo Go で体験検証

- iPhone アプリの画面、入力、履歴、読み上げを作る。
- SSH 実接続はまだ行わず、モック応答または手動で用意したサンプル応答を使う。
- 散歩中に、テキスト入力と読み上げが実用に耐えるかを確認する。

### Phase 2: Mac 側ブリッジで接続検証

- Mac 側に小さなローカル HTTP または WebSocket ブリッジを置く。
- iPhone アプリは Tailscale 経由でブリッジに接続する。
- ブリッジが tmux と Codex CLI を操作する。
- iPhone アプリ側は Expo Go のまま、SSH ネイティブ実装を避けて検証する。

### Phase 3: Development Build で直接接続を検証

- SSH を iPhone アプリから直接扱う必要がある場合に進む。
- Expo Go ではなく Development Build を使い、必要なネイティブライブラリを含める。
- Apple Developer Program への加入や実機配布の扱いを判断する。

## TTS の見通し

Expo の `expo-speech` は Expo Go に含まれているため、基本的な TTS は Expo Go で検証できる。

ただし、散歩中の雑談体験として十分かどうかは別問題である。長い応答の分割、読み上げ中断、読み上げ速度、声質、日本語の自然さ、iPhone のサイレントモード時の挙動を確認する必要がある。Expo 公式ドキュメントでは、iOS 実機のサイレントモード中は `expo-speech` が音を出さないと案内されている。

irodori-TTS を使う場合は、iPhone 側で直接動かすのか、Mac 側で音声生成するのかを検討する。iPhone 側でネイティブ連携が必要な場合は、Expo Go では難しい可能性が高い。

## 主な懸念

| 懸念 | 影響 | 初期対策 |
| --- | --- | --- |
| ブリッジサーバーの認証 | Tailscale 内とはいえ、Mac 上の Codex CLI を遠隔操作できる入口になる | 最初から共有トークン認証を入れ、Tailscale のアドレスからのみ受ける |
| コマンド注入 | iPhone から渡した文字列が tmux や shell コマンドとして意図せず解釈される | `tmux send-keys` に渡す入力を厳密に扱い、shell 文字列連結を避ける |
| Codex CLI の状態判定 | 実行中、完了、入力待ち、エラーを安定して判定できない可能性がある | 最初は手動更新または一定間隔の `tmux capture-pane` で始め、最終的には `tmux -CC` の control mode を検証する |
| 出力の差分取得 | 毎回ペイン全体を取得すると、重複表示や読み上げ重複が起きる | 最初は全文表示で検証し、後で control mode の `%output` 通知を使った差分管理を検討する |
| 長文応答の読み上げ | 散歩中に長い応答を聞く体験が破綻する可能性がある | 読み上げ対象を要約または最終結果に絞る |
| バックグラウンド動作 | iPhone をロックした状態で通信や読み上げが期待どおり続かない可能性がある | 初期段階では画面表示中の利用に限定して検証する |
| Mac 側プロセス管理 | ブリッジサーバーが落ちると接続できない | 最初は手動起動、後で `launchd` などで常駐化する |
| 秘密情報の露出 | Codex CLI の出力や tmux ペインに秘密情報が含まれる可能性がある | ログを最小化し、ブリッジ側で不用意に出力全文を永続化しない |
| Tailscale 前提 | Tailscale が切れていると利用できない | 対象外としつつ、アプリ上では接続失敗を分かりやすく表示する |

## 初期検証で優先して潰すリスク

1. Expo Go から Mac 側ブリッジへ Tailscale 経由で到達できるか。
2. ブリッジから tmux セッションに安全にプロンプトを送れるか。
3. `tmux capture-pane` で Codex CLI の応答を、アプリで読める形にできるか。
4. `tmux -CC` の control mode で、出力差分をより安定して扱えるか。
5. `expo-speech` の日本語読み上げが散歩中の利用に耐えるか。
6. 読み上げが無い場合でも、開発指示アプリとして価値があるか。

## 初期方針

- 最初は Expo Go で始める。
- 最初から Apple Developer Program の費用を前提にしない。
- 初期検証では、Expo Go で動かすことを優先する場合に「Mac 側ブリッジ経由」を有力候補にする。
- 最終構成としては、SSH 直接接続がより単純になる可能性を残す。
- TTS は `expo-speech` で先に検証する。
- irodori-TTS は、品質要件が見えた後に採用可否を判断する。

## 判断が必要な点

- Expo Go 期間中は、SSH 直接接続を諦めて Mac 側ブリッジを許容するか。
- 最初の価値検証を、AI 雑談に寄せるか、開発指示に寄せるか。
- TTS の自然さをどの程度まで必須にするか。
- Development Build や Apple Developer Program へ移行する判断基準をどこに置くか。
