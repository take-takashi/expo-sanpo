# Remote Codex Session

## 目的

iPhone アプリから母艦の Mac 上の tmux セッションに接続し、Codex CLI にプロンプトを送信して結果を確認できるようにする。

散歩中の利用では、AI との雑談、開発指示、調査依頼を主な入力内容として想定する。

## 前提

- ユーザーは Tailscale により、iPhone から母艦の Mac へ到達できる。
- VPN の設定、接続、運用はこの機能の対象外である。
- 母艦の Mac には SSH 接続できる。
- 母艦の Mac には tmux と Codex CLI が導入されている。

## 基本操作

1. ユーザーは iPhone アプリで接続先の Mac または Mac 側ブリッジを選ぶ。
2. アプリは Mac 側ブリッジに接続する。
3. アプリは指定された tmux セッションを作成または再利用する。
4. Mac 側ブリッジは tmux 上で Codex CLI を起動または再利用する。
5. ユーザーはアプリからプロンプトを入力する。
6. アプリはプロンプトを tmux 上の Codex CLI に送る。
7. アプリは Codex CLI の出力を取得し、画面に表示する。
8. 読み上げが有効な場合は、アプリは結果を TTS で読み上げる。

## ユースケース

### 散歩中の AI 雑談

ユーザーは散歩中に、考えごと、アイデア、悩み、作業方針などを自然文で入力し、Codex CLI からの応答を確認する。

読み上げが有効な場合は、アプリは応答を音声で読み上げる。読み上げが無効または未実装の場合でも、アプリは応答をテキストで表示し、会話を継続できる。

### 散歩中の開発指示

ユーザーは散歩中に、実装、調査、リファクタリング、ドキュメント更新などの指示を Codex CLI に送る。

読み上げが難しい場合でも、このユースケースは成立する。アプリは少なくとも、プロンプト送信と結果確認を提供する。

## 期待挙動

- アプリは、SSH 接続が切れても tmux セッション上の Codex CLI 実行を維持できる。
- アプリは、再接続後に既存の tmux セッションへ戻れる。
- アプリは、ユーザーが入力したプロンプトを Codex CLI に送信できる。
- アプリは、Codex CLI の結果をユーザーが読める形で表示できる。
- 会話履歴の正本は Mac 側ブリッジが保持または再構成する。
- アプリは、表示用の一時キャッシュを持ってもよいが、会話履歴の正本にはしない。
- アプリは、TTS が有効な場合に結果を音声で読み上げられる。
- TTS が無い状態でも、散歩中に開発指示や調査依頼を送れる。
- AI 雑談は TTS がある場合に体験価値が高いが、TTS を必須要件にはしない。

## 対象外

- 汎用ターミナルエミュレータとしての完全な ANSI 表示や対話操作。
- Tailscale のセットアップや状態管理。
- Mac 側の Codex CLI のインストール自動化。

## 初期 PoC

現時点では、iPhone アプリから Mac 側ブリッジへ接続し、`POST /sessions` でメモリ上のセッションを作成し、`POST /sessions/:id/prompts` で user message と mock assistant message を追加し、`GET /sessions/:id/messages` でメッセージ一覧を取得する。

tmux との最小接続検証では、`EXPO_SANPO_PROMPT_DRIVER=tmux` を指定した Mac 側ブリッジが tmux 上の `cat` セッションへ入力し、`tmux capture-pane` の結果を assistant message として返す。Codex CLI の最小検証では、`EXPO_SANPO_PROMPT_DRIVER=codex` を指定した Mac 側ブリッジが tmux 上に `codex --no-alt-screen` を起動し、`tmux send-keys` で prompt を入力し、設定された送信キーを送ってから、送信前後の `tmux capture-pane` 差分を assistant message として返す。送信キーはデフォルトで Option+Enter 相当の `Escape` + `Enter` とし、Mac 側の Codex CLI 設定に合わせて差し替えられる。現時点では完了判定は未実装であり、送信後に短く待ってから画面 capture を返す。

## 未確定事項

- tmux セッション名の命名規則。
- Codex CLI が未起動の場合に自動起動するか、ユーザーに確認するか。
- プロンプト送信時のキー列をアプリ設定、Mac 側設定ファイル、環境変数のどれで管理するか。
- 出力取得は初期検証では `tmux capture-pane` を使い、最終的には `tmux -CC` の control mode を使うか。
- Codex CLI の完了判定をどのように行うか。
- TTS の読み上げ対象を最終応答だけにするか、途中出力も対象にするか。
- 雑談用セッションと開発作業用セッションを分けるか。
- 読み上げに適した短い応答を Codex CLI に促すプロンプト設計が必要か。
- PoC の会話履歴は Mac 側ブリッジのメモリに保持し、永続化は後で再検討する。
